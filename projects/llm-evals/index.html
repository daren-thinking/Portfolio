基础概念
Eval 是什么？
在动手搭 Eval 系统之前，先把概念想清楚。很多人走弯路，是因为对"评估"的理解本身就错了。
核心定义

Trace：评估的基本单位

Trace 是从用户发出第一条消息，到系统给出最终回复的完整记录——包括所有中间步骤、工具调用、检索结果、多轮对话上下文。

在 LLM 产品里，一个 Trace 可能包含：用户输入 → 检索召回 → Prompt 构建 → 模型推理 → 工具调用 → 最终输出。每一个环节都是可以出问题的地方。

Eval：基于 Trace 的判断

Eval = 基于 Trace，判断系统在这次交互里有没有失败。

注意：Eval 不是一个工具，不是一个平台，不是一个 Dashboard。Eval 是一个判断活动——由人、由规则、或由另一个模型来执行这个判断。

Eval vs 传统软件测试
❌ 传统软件测试

输入确定，输出确定。测试是验证逻辑正确性，有明确的 Pass/Fail 标准，可以 100% 自动化。

✅ LLM Eval

输出是概率性的，没有唯一正确答案。"好的回答"是主观判断，需要人工建立评判标准，自动化只能辅助，不能替代。

⚠️ 常见误区

不能把 LLM Eval 理解为"测试 AI 对不对"。应该理解为"在真实条件下，AI 是否达到了用户期望的结果"。"对不对"是技术视角，"是否达到期望"是产品视角——两者在 LLM 场景里经常不一致。

Eval 的三种执行方式
方式 执行者 优势 局限 适用场景
规则/断言 代码 快、稳定、可重现 只能覆盖可形式化的约束 格式验证、禁止词检查、长度限制
人工评估 人 最准确、能捕捉细微差异 慢、贵、不可扩展 建立 Ground Truth、验证自动化
LLM-as-Judge 另一个 LLM 可扩展、能处理复杂判断 必须对齐人类判断才可信 大规模自动评估（需先人工校准）
💡 关键原则

三种方式不是替代关系，是叠加关系。任何成熟的 Eval 系统都三者并用。人工评估是基础，不可跳过；LLM-as-Judge 是人工评估的放大器，不是替代品。

基础概念
为什么必须做 Eval？
不是因为"最佳实践"，而是因为没有 Eval，AI 产品无法突破某个瓶颈——而且你不知道自己卡在哪。
没有 Eval 会发生什么

AI 产品在早期往往进展顺利：Prompt Engineering 快速出效果，Demo 好看，团队信心满满。但到某个阶段，进展会突然停滞：

  • 修复一个失败模式，另一个随之出现——打地鼠效应
  • 对 AI 的整体效果，只有"感觉"，没有数据
  • Prompt 越改越长，试图覆盖越来越多的 Edge Case
  • 无法判断一次 Prompt 修改是否真的让整体变好了

这是几乎所有 AI 产品都会经历的平台期。突破它的唯一方法是建立系统化的评估体系。

Eval 创造的飞轮效应
✅ Eval 系统的价值不止于评估本身

一个好的 Eval 系统，同时就是你的：调试基础设施（快速定位失败根因）、精调数据引擎（高质量标注数据直接用于 Fine-tune）、数据合成工具（Eval 判断可以过滤合成数据质量）。

Eval 能力 额外获得的能力
Trace 日志 + 搜索过滤 调试能力：快速找到失败的根因是模型、RAG 还是代码 Bug
人工标注（Pass/Fail + Critique） Fine-tune 数据：直接用于模型精调，质量有保证
LLM-as-Judge + 失败分类 合成数据过滤：生成大量合成数据，用 Judge 过滤低质量样本
CI/CD 集成的 Unit Tests 回归防护：每次代码变更自动验证，防止好不容易修复的问题复现
迭代速度决定成败

AI 产品的成功，和软件工程一样，核心取决于迭代速度。迭代速度需要三个条件同时具备：

  1. 评估质量的能力（Eval 系统）
  2. 调试问题的能力（日志 + 工具）
  3. 改变系统的能力（Prompt Engineering、Fine-tune、代码）

大多数团队只有第三条，缺前两条。结果是改了很多，但不知道是否变好，也不知道为什么失败。

Trace：评估的基本单位
理解 Trace 是什么，如何记录，以及如何让查看 Trace 变得没有摩擦——这是 Eval 系统的基础设施。
Trace 的结构

Trace 是一次完整交互的全量记录。对于一个 Agent 系统，一个 Trace 可能包含：

消除查看 Trace 的摩擦
💡 核心原则

Trace 查看工具的设计原则只有一条：把理解一次交互所需的所有信息放在同一个屏幕上，不需要跳转，不需要打开另一个系统。任何摩擦都会导致你少看数据，而少看数据就意味着少发现问题。

一个好的 Trace 查看工具应该有什么

  • 当前交互涉及哪个 Feature 和 Scenario
  • 是真实用户输入还是合成测试数据
  • 快速过滤：按 Feature、Scenario、用户类型、时间段
  • 相关业务系统的直接链接（CRM、数据库、日志系统）
  • Pass/Fail 标注入口，不需要离开界面

工具选择

不需要买贵的工具。LangSmith（不必须用 LangChain）是不错的起点。更好的做法是用 Gradio / Streamlit / Shiny 在一天内自己搭一个，因为每个业务的信息需求都不同，定制工具比通用工具更高效。

哪些 Trace 值得重点看
优先级类型原因
高优先 被用户明确反馈差的对话 真实失败，直接影响用户体验
高优先 多次重试或对话特别长的 用户在挣扎，系统没有解决问题
中优先 触发异常或错误的对话 技术失败，可能有系统性原因
日常 随机抽样的正常对话 维持对系统整体状态的感知

Error Analysis：整个 Eval 体系的起点
不是工具，不是基础设施。是人工读数据，是建立业务直觉的过程。这是 Eval 里最重要的活动，没有之一。
💡 为什么 Error Analysis 最重要

Error Analysis 回答的是"我需要评估什么"，而不是"如何评估"。很多团队跳过这一步，直接用现成的 Eval 指标，结果测的是和自己业务完全无关的东西——指标很好看，产品一样烂。

Error Analysis 的四步流程
1
Open Coding（开放编码）
人工读 Trace，用自由格式写失败笔记。不预设分类，不套模板。就像写日记一样，看到什么问题就写什么。这一步必须自己做，不能外包给 LLM——这是建立业务直觉的过程，这个过程本身比结果更有价值。

示例笔记："用户问了一个有歧义的问题，模型选了最常见的解读，但用户的上下文说明他想要的是另一种。" / "模型的回答格式完全正确，但忽略了一个关键约束条件。"
2
Axial Coding（轴向编码）
把 Open Coding 的笔记归类，形成"失败分类体系"，统计每类出现的次数。这时候可以用 LLM 辅助——把你写的 30-50 条笔记给 LLM，让它帮你归类和总结。但 LLM 的归类结果必须人工校验。

示例失败分类：歧义输入处理失败 / 约束条件遗漏 / 知识边界外幻觉 / 多步推理链断裂 / 格式不符合预期……
3
迭代直到理论饱和
持续看新 Trace，继续更新分类体系。当新的 Trace 不再触发新的失败类别时，说明你的分类体系已经覆盖了当前的主要失败模式——这叫理论饱和，是停止迭代的信号。
4
基于分类体系设计 Eval
现在你知道要评估什么了。每一个失败类别，都可以对应设计一个具体的 Eval：断言（Unit Test）或 LLM Judge 的评判标准。从最高频的失败类别开始，优先覆盖。
数量建议

至少读 100 条 Trace 才能开始建立系统性认知。在这之前，你看到的只是随机样本，不足以形成可靠的失败分类。

一个实用的停止标准：读完新的 Trace 不再觉得有什么新东西——这比看一个固定数字更准确。

LLM 在 Error Analysis 中能做什么、不能做什么
❌ 不能交给 LLM

• 初始的 Open Coding（必须自己读，这是建立业务直觉的过程）
• 验证失败分类体系的准确性
• 根因分析（为什么这类失败反复出现）
• Ground Truth 标注（用于校准 LLM Judge 的数据必须人工标）

✅ 可以用 LLM 辅助

• 将 Open Coding 笔记归类（在你写了 30-50 条之后）
• 把新的失败案例映射到已有的失败分类
• 生成更多覆盖特定失败类别的测试用例
• 提示词改进建议（人工审核后再用）

Unit Tests：快速反馈的基础层

Unit Tests 是用代码写的断言（Assertions），在每次代码变更时自动运行。它们不能测所有东西，但能快速捕捉可形式化的失败。

设计原则

  • 按 Feature + Scenario 组织，不要全部混在一起
  • 断言要尽量具体，避免只测"有没有返回结果"
  • 通过率不需要 100%——多少可接受是产品决策，不是技术决策
  • 持续更新：每次在线上发现新的失败模式，就转化为新的 Unit Test
# 示例：客服 AI 的 Unit Tests def test_intent_recognition_return(): response = ai.chat("我想退款") assert response.intent == "return_request" def test_no_competitor_mention(response): # 通用测试：不得提及竞品 competitors = ["竞品A", "竞品B"] for competitor in competitors: assert competitor not in response.text def test_policy_query_contains_facts(): response = ai.chat("七天无理由退换货政策是什么") assert "7天" in response.text or "七天" in response.text assert response.token_count < 500 # 不应该过于冗长
⚠️ 重要：合成测试数据

不要等到有真实用户数据才开始写 Unit Tests。可以用 LLM 生成合成的测试输入，覆盖你能预想到的各种 Scenario。生成方法：给 LLM 详细的场景描述，让它生成 50-100 条不同的用户输入。

Human & Model Eval：质量判断的核心层

Unit Tests 能覆盖的范围有限——很多质量问题无法用断言表达。Human & Model Eval 是处理这类判断的方法。

人工评估的流程

  1. 记录 Trace（所有对话都要记录）
  2. 消除查看 Trace 的摩擦（一屏看到所有相关信息）
  3. 人工做 Pass/Fail 判断 + 写 Critique
  4. 分析标注结果，识别系统性失败模式

何时引入 LLM-as-Judge

在有了足够的人工标注数据（至少 30-50 条）之后，可以尝试用 LLM 自动化部分判断。但必须持续验证 LLM Judge 和人类判断的一致性，不一致时以人类判断为准。

💡 定性信号：什么情况下该扩大人工评估

当你发现 LLM Judge 的结论和你直觉不符时；当产品发生重大迭代时；当进入新的用户场景时——这些时候都应该重新做一轮人工评估，重新校准 Judge。

A/B Testing：验证真实用户行为

A/B Testing 是最贵的评估方式，也是最接近真实业务价值的评估方式。它验证的不是"AI 的回答质量"，而是"这个变化是否真正改变了用户行为或业务结果"。

适用时机

  • 产品已经有稳定的用户基础
  • Level 1 和 Level 2 的 Eval 已经稳定运行
  • 有明确的业务指标可以衡量（转化率、留存率、任务完成率）
  • 做了重大的产品或模型变更，需要验证真实效果

AI 产品 A/B Test 的注意点

  • AI 输出的随机性更大，需要更大的样本量才能达到统计显著性
  • 要分清楚"AI 质量提升"和"用户行为变化"——两者不总是同向
  • 短期用户行为可能反映新鲜感，而非真实价值

核心方法论
Pass/Fail vs 评分制：为什么二元判断更好
这是一个反直觉的结论，但有扎实的依据。用 1-5 分评估 AI 输出质量，几乎总是个错误。
为什么 1-5 分不好用
❌ 1-5 分评估制的问题

• 没有行动指南：3 分和 4 分，团队应该做什么不同的事？
• 主观差异大：不同标注者对"3分"的理解不同，数据不可汇总
• 倾向于中间值：标注者会逃避判断，大量结果堆积在 2-4 分
• 相关性弱：每次分析域专家的判断数据，1-5 分指标往往和真正的质量不相关
• 维度越多越糟：同时在 8 个维度上打 1-5 分，既不可靠也不可行

✅ Pass/Fail 的优势

• 强迫做判断：二元选择迫使标注者思考什么是真正的成功条件
• 行动清晰：Fail 就是需要改进，Pass 就是基线达到
• 迫使标准外显化：通过做 Pass/Fail 判断，专家会逐渐说清楚标准是什么
• 速度更快：同样的时间能标注更多数据
• 更容易对齐：两个标注者对 Pass/Fail 的分歧比对分数的分歧更容易解决

Pass/Fail + Critique 的组合

Pass/Fail 不是简单粗暴的二元。配合 Critique（评价文字），可以同时得到清晰的判断和丰富的细节信息。

Critique 的作用：

  • 捕捉"大体上 Pass，但有改进空间"的细节
  • 解释为什么 Fail，指向具体的改进方向
  • 这些 Critique 后来会成为 LLM Judge Prompt 的 Few-shot 示例
用户输入AI 回答判断Critique
"我想取消明天的航班" "您的航班已取消，确认邮件已发送。" Pass 完成了用户请求。但应该在取消前要求确认，避免误操作。虽然有这个缺陷，核心目标完成，所以 Pass。
"我登录不进去，提示密码错误" "请点击'忘记密码'重置密码。" Fail 给了一个通用解决方案，没有询问是否账号被锁定，没有表现出对用户遭遇的理解，体验差。Fail。
如果需要追踪细粒度改进怎么办

把大指标拆成多个独立的 Pass/Fail 检查，而不是用评分。

例如，不要用"回答质量 1-5 分"，而是拆成：

  • ✅/❌ 是否包含了用户需要的核心信息
  • ✅/❌ 是否没有提到不存在的内容
  • ✅/❌ 语气是否匹配当前场景
  • ✅/❌ 是否在合理的长度范围内

这样你有 4 个可以追踪的指标，每一个都是 Pass/Fail，都有清晰的改进方向。

领域专家与数据标注：谁来判断"好"
Eval 系统里最常被忽视的问题：谁来标注数据？答案比你想的简单，也比你想的重要。
Benevolent Dictator（仁慈独裁者）模型
💡 核心建议

找一个熟悉你的用户和业务的领域专家，由他/她来做最终质量判断。不需要多人投票，不需要共识机制。一个真正懂业务的人，胜过五个不懂的人投票。

为什么不用多人标注？

  • 多人标注会带来分歧，而分歧不是数据，是噪音
  • 解决分歧需要额外流程，拖慢迭代
  • 如果评判标准不统一，多人标注比单人标注还不可靠

Benevolent Dictator 的要求

  • 深度理解目标用户（最好自己就是代表用户）
  • 清楚业务目标和约束
  • 能够清晰解释自己的判断（不是凭感觉）
  • 愿意投入时间：不需要很多，但需要持续参与

客服 AI客服运营负责人
教育 AI 工具有经验的教师 / 课程设计师
AI 产品（你的场景）深度理解用户且有产品判断力的产品负责人
不要外包标注
⛔ 反模式

把数据标注外包给不了解业务的人，是最常见的 Eval 失败原因之一。外包的人按通用标准打分，但你的业务有自己独特的"好"的定义。外包标注数据质量低，还会误导你的 LLM Judge 训练方向。

唯一例外：纯机械性任务——比如验证邮箱格式是否正确、数字是否在合理范围内。

进阶专题
LLM-as-a-Judge：Critique Shadowing 方法论
如何建立一个能自动化评判 AI 输出质量、且与人类判断高度一致的 LLM 评判系统。
⚠️ 先决条件

在开始建 LLM Judge 之前，你必须已经完成了 Error Analysis，有了 30 条以上的人工 Pass/Fail + Critique 标注。没有这些，LLM Judge 没有对齐的基础，建了也是错的。

Critique Shadowing：7 步流程
1
找到 Principal Domain Expert
确定 Benevolent Dictator，他的判断是 Ground Truth。
2
构建有代表性的数据集
覆盖真实用户场景（Feature × Scenario × Persona），至少 30 条，越多越好。真实数据 + 合成数据结合。
3
人工做 Pass/Fail + Critique
专家对每个交互写详细的评价，解释为什么 Pass 或 Fail。这些 Critique 就是你 LLM Judge Prompt 的素材。
4
修复明显的系统错误
先把人工评估发现的明显失败修掉，再建 Judge。Judge 是为了扩大你的感知范围，不是为了掩盖已知问题。
5
迭代构建 LLM Judge Prompt
把人工 Critique 作为 Few-shot 示例，写 Judge Prompt。跑数据，对比 Judge 的判断和人工判断，迭代直到一致性足够高。
6
Error Analysis（再次）
现在有了自动化 Judge，可以大规模分析失败模式了。把分类结果反馈给专家验证，迭代分类体系。
7
创建专项 Judge（如需要）
如果某个特定的失败类型需要更精细的判断，单独为它创建一个专项 Judge，而不是让一个通用 Judge 负责所有事情。
Judge Prompt 的关键要素
# LLM Judge Prompt 结构模板 You are a [领域] 质量评估专家，负责判断 AI 的回答是否达到预期效果。 ## 评判标准 {{guidelines}} # 从 Critique 中提炼出的具体标准 ## 示例评判（Few-shot，直接来自人工标注） <example-1> <input>[用户输入]</input> <response>[AI 回答]</response> <critique> 判断: Pass 原因: [从人工 Critique 原文改写] </critique> </example-1> <example-2> # 一个 Fail 的例子，且要求 critique 足够详细 </example-2> ## 待评估的交互 <input>{{user_input}}</input> <response>{{ai_response}}</response> ## 输出格式 { "outcome": "pass" | "fail", "critique": "[详细解释]" }
常见的 LLM Judge 错误
  • 用 off-the-shelf Judge（如直接用 GPT 的通用评估 API）：这些 Judge 不知道你的业务标准，输出结果不可信
  • Critique 太简短："回答不够准确" 这样的 Critique 不够用，要具体到"哪里不准确，为什么不对"
  • 不验证一致性：建完 Judge 之后不持续对比人类判断，任凭 Judge 漂移
  • Judge 用的模型太弱：评判是推理密集任务，用强模型（比生产用的模型更强），可以接受慢一点
  • Criteria Drift：评判标准是在做判断的过程中逐渐明确的，不可能在开始前就完全定义好——接受这个事实，持续迭代标准

进阶专题
数据集构建：Feature × Scenario × Persona
好的 Eval 数据集需要有代表性——不是随机采样，而是有意识地覆盖你的 AI 可能面对的各种情况。
三个维度定义覆盖范围
维度定义示例
Feature AI 的具体功能模块 意图识别、政策查询、退款处理、情绪识别
Scenario AI 需要处理的情况类型 歧义输入、无匹配结果、多个结果、系统错误、信息缺失
Persona 代表性的用户类型 新用户、专业用户、非母语用户、情绪激动的用户
⚠️ 注意

这个三维框架不是通用标准，是一种组织思路。如果你的场景用户不直接和 AI 交互（比如自动化流水线），Persona 维度可能不适用。根据你的实际业务定义有意义的维度。

合成数据生成

不要等到有足够的真实用户数据才开始 Eval。合成数据可以作为很好的起点，覆盖你预想的各种组合。

重要原则：只用 LLM 生成用户输入，不用 LLM 生成 AI 回答。把合成的用户输入喂给你自己的 AI 系统，让系统生成回答，然后对这些真实的 AI 回答进行评估。

# 合成用户输入的 Prompt 模板 生成 30 个不同的用户输入，满足以下条件： - Feature: [具体功能，如"退款申请"] - Scenario: [场景类型，如"用户没有提供订单号"] - Persona: [用户类型，如"情绪激动的用户"] 要求： - 用自然语言，不同长度和表达方式 - 包含真实用户会犯的错误（错别字、不完整信息） - 输出 JSON 数组格式 示例输出： [ "我昨天买的东西想退，但是我找不到订单号了", "这东西质量太差了！！！必须给我退款", ... ]

"LLMs 在生成多样化的用户输入方面出乎意料地好。这听起来像蛇在吃自己的尾巴，但它确实有效——ship it。"

— Bryan Bischof, Head of AI Engineering at Hex
数据集规模建议
  • 最小起步：每个 Feature × Scenario 组合至少 1 个样本
  • 实用目标：30-50 条人工标注数据作为 LLM Judge 的校准集
  • 何时停止扩大：当新的数据不再触发新的失败模式（理论饱和）
  • 动态更新：每次上线发现新的失败，就加入数据集；数据集是活的，不是一次性的

进阶专题
RAG 的评估：检索和生成要分开看
RAG 系统有两个可能失败的地方，失败原因不同，解法也不同。混在一起评估等于什么都看不清楚。
RAG 的两层失败
📚 检索失败

知识库里有正确答案，但没有检索到；或者检索到了，但质量差的文档把好的文档稀释了。

症状：回答模糊、不完整，或者"根据提供的信息，我无法回答"。
解法：改进 Embedding 模型、调整检索参数、优化知识库结构。

🤖 生成失败

检索到了正确内容，但模型没有正确利用，或者超出检索内容进行了幻觉扩展。

症状：回答里有检索内容，但有错误解读；或者在检索内容基础上"发挥"了不存在的内容。
解法：改进 Prompt 对上下文的利用指令、增加引用约束。

RAG Eval 框架
评估维度 问题 评估方法
检索召回率 正确答案所在的文档有没有被召回？ 构建包含 Ground Truth 文档 ID 的测试集，检查是否在 Top-K 结果里
检索精确率 召回的文档有多少是有用的？ 人工或 LLM Judge 评估每个检索结果与 Query 的相关性
答案忠实度 生成的回答有没有超出检索内容？ LLM Judge 判断：回答中的每个声明是否都能在检索内容中找到支持
答案完整度 用户需要的信息是否都包含在回答里？ 对比 Ground Truth 答案，人工或 LLM Judge 判断
💡 最重要的一步

在分析 RAG 失败时，先做一个简单判断：如果把正确的检索内容直接给模型，它能给出正确答案吗？如果能，问题在检索；如果还是不能，问题在生成。这个二分法能节省大量调试时间。

知识库质量的重要性

很多 RAG 系统的问题不在检索算法，而在知识库本身：文档质量参差不齐，重复内容多，更新不及时，缺少对特定业务场景的覆盖。

对于业务场景有明确约束的场景（如游戏 AI 必须符合游戏世界观设定，不能混入其他游戏的内容），知识库的准确性和覆盖度比检索算法更重要。

这也是为什么 QA 对、业务知识库的数据体量和质量，在 AI 应用里仍然非常重要——它是 AI 能够在边界内准确回答的基础。

CI/CD Eval生产环境监控

Guardrail vs Evaluator
🛡️ Guardrail

实时拦截：在 AI 输出之前或之后立即执行判断，阻止有害内容触达用户。

要求：低延迟（毫秒级），高精确率（宁可误报，不能漏报）。

示例：内容安全检查、PII 泄露检测、竞品提及检查。

📊 Evaluator

事后分析：在交互完成后评估质量，用于改进系统，不直接影响用户体验。

要求：高质量（准确反映质量），可以慢一点。

示例：回答相关性评估、任务完成度判断、用户满意度预测。

生产 Error Analysis 的频率建议
  • 每次重大变更后：Prompt 大改、模型升级、新 Feature 上线——必须重新做一轮完整的 Error Analysis
  • 每周常规：随机抽取 10-20 条真实 Trace 阅读，重点关注异常对话（特别长、多次重试、有用户投诉标记的）
  • 每月系统性回顾：系统化分析一段时间内的失败分布，更新失败分类体系

一个实用的停止标准：直到你不再从新数据里学到新东西。这个信号比固定的时间频率更准确。

💡 反直觉：100% Pass Rate 是警告信号

如果你的生产 Eval Pass Rate 达到 100%，很可能是你的 Eval 不够难，在给自己制造虚假安全感。70% 的 Pass Rate 可能更真实地反映了系统的实际水平，也是更有价值的工作起点。

7 个最贵的 Eval 错误
❌ 错误 1：使用 Generic 评估指标

BERTScore、ROUGE、Cosine Similarity……这些指标测的是抽象的文本相似度，和你的业务目标没有直接关系。高分不代表产品好用，低分不代表产品差。停止使用它们，除非你的场景真的就是文本相似度（比如 RAG 检索质量初步筛选）。

❌ 错误 2：Eval-Driven Development（提前定义所有 Eval）

在实际用户数据出现之前，你无法预判 LLM 会以什么方式失败。提前写好所有 Eval，测的是你想象中的失败，不是真实的失败。正确做法：先上线（哪怕是小规模），看真实数据，再设计 Eval。
例外：有非常明确的硬性约束（"绝对不能提到竞品"），可以提前写。

❌ 错误 3：跳过人工评估，直接上自动化

LLM Judge 是人工判断的放大器，不是替代品。没有人工标注的 Ground Truth，LLM Judge 对齐的是什么？是对什么、基于什么标准？必须先建立人工判断基线，才能构建可信的自动化评估。

❌ 错误 4：外包数据标注给不了解业务的人

外包标注者按通用标准打分，但你的业务有独特的"好"的定义。他们的标注不只是质量差，而且会系统性地误导你的 LLM Judge 校准方向——是负的价值，不只是零价值。

❌ 错误 5：用 1-5 分评估维度来堆砌指标

"业务方说要评估 8 个维度，每个维度打 1-5 分。" 如果有人这么说，说明他们还不知道自己在找什么。由 Benevolent Dictator 做 Pass/Fail 判断，从数据里找到真正重要的维度，而不是事先假设 8 个都重要。

❌ 错误 6：不看数据，只看指标仪表盘

仪表盘告诉你数字，数字告诉你有问题，但不告诉你问题是什么。任何 Eval 系统都不能替代直接阅读失败案例。人工阅读数据是保持对系统真实状态感知的唯一方式，也是发现新问题的起点。

❌ 错误 7：100% Pass Rate 就停止迭代

Pass Rate 100% 几乎总是说明你的 Eval 不够难，不是说明你的系统真的完美。持续更新测试用例，纳入新发现的失败案例，让你的 Eval 持续保持挑战性。

💡 为什么 PM 应该主导 Eval 标准

技术团队能告诉你模型的通用能力，但只有你知道：用户在什么条件下提问、什么叫足够好的答案、什么程度的失败是可以接受的、什么样的失败会毁掉整个体验。这些判断没有人比 PM 更有资格做。

PM 在 Eval 中的具体职责
1
定义"好"的标准
明确说出用户在什么情况下会认为 AI 回答足够好，什么情况下会失望。这个标准不能只停留在"直觉"里，需要在 Error Analysis 过程中被显式化出来。PM 做的每一次 Pass/Fail 判断，都是在把这个标准从隐性变成显性。
2
把产品拆成子任务，逐一定层
不要整体评估"这个 AI 行不行"，要按用户路径把产品拆成细粒度子任务，每个子任务独立评估它属于哪一层边界（模型上限 / 应用方式 / 工程架构）。这个拆解过程，PM 是最有能力做的。
3
主导 Bad Case 库的建立和分类
每周阅读最差的 20 条 Trace，按失败原因归类。这个工作是 PM 和工程师都能做的，但 PM 的视角更接近用户——能发现工程师看不到的失败（比如"技术上没错，但用户体验很差"）。
4
基于层次判断，指导架构投入方向
当知道每个子任务的失败属于哪一层，就能给出清晰的优先级：工程架构层的问题先修（投入产出比最高）；模型上限层的问题考虑换模型或降级方案；应用方式层的问题交给提示词优化。

❓ 常见问题 FAQ
实战中最常被问到的 6 个问题，直接给答案
📌 这些问题从哪里来？

这 6 个问题是 Hamel Husain 和 Shreya 在教授 700+ 工程师和 PM 做 AI Evals 课程中，被反复提问的核心问题。每个答案都经过大量真实项目验证。

Q1：当输出结果需要主观判断时，我该如何测试应用程序？

这是最常见的困惑，也是整个 Eval 体系设计的出发点。主观判断 ≠ 无法测试，关键是把"主观"变成"可操作的标准"。

1
找到你的 Benevolent Dictator（仁慈独裁者）

一个真正理解用户和业务的人。他/她的判断就是你的"黄金标准"。多人投票解决不了主观问题，反而会引入噪音。你需要的是判断标准的一致性，不是民主。

2
用 Pass/Fail + Critique 代替评分

不要问"这个答案几分？"，而是问"这个答案，用户会不会满意？为什么？"。强制二元判断，然后写下具体原因。这个过程会逼迫你把模糊的"好感觉"变成可复现的标准。

3
先做 Error Analysis，再定标准

在看真实数据之前，你其实不知道"主观"在哪里。读 50 条真实对话，你会发现大多数"失败"其实是有规律的：格式问题、遗漏关键信息、语气不对……这些都可以变成明确的 Pass/Fail 标准。

4
对真正主观的部分：接受不确定性，记录分歧

有些判断确实因人而异。这时候不要强求统一，而是记录下"什么情况下标注者会有分歧"，这本身就是对产品边界的重要洞察。

💡 核心原则

主观判断不是 Eval 的障碍，而是 Eval 必须解决的核心问题。你不能因为"主观"就放弃量化——量化的过程本身就是澄清产品标准的过程。

Q2：如果我更改了提示（Prompt），如何知道不会破坏其他东西？

这是 Prompt 工程的经典困境——改好一个地方，不知道有没有搞坏别的地方。解法是建立分层的回归测试体系。

第一层：Unit Tests
第二层：人工抽样对比
第三层：A/B Test

自动化回归测试（每次 Prompt 变更必跑）

把之前发现过的失败案例都写成 Unit Test。每次改 Prompt，先跑 Unit Tests——如果任何已知失败重新出现，立刻阻止上线。成本极低，应该每次都做。

  • 把 每一个历史 Bug 都写成测试用例
  • 包含边界情况、已知的坏输入
  • 通过 = 没有回退，不通过 = 先别上

人工 Side-by-Side 对比（重大变更前做）

取 20-50 条有代表性的生产 Trace，同时跑旧版 Prompt 和新版 Prompt，让领域专家逐条比较哪个更好。Unit Tests 只能防已知失败，Side-by-Side 才能发现整体质量变化。

  • 选取覆盖不同场景的多样化样本
  • 让专家盲评（不知道哪个是新/旧）
  • 记录每道题的判断理由

A/B Test（有充分信心时的最终验证）

成本最高，不要对每个 Prompt 变更都做。当前两层都通过了，你有相当信心新版更好，才考虑上 A/B Test 验证真实用户行为。

  • 需要足够的流量才有统计意义
  • 定义好核心业务指标（不只是 Eval Pass Rate）
  • 设好提前终止条件
Q3：我应该把工程精力集中在哪些方面？需要测试所有东西吗？

不需要测试所有东西。关键是把精力花在 ROI 最高的地方。

✅ 优先做
  • Error Analysis — 永远是第一位，没有之一
  • 高频 × 高影响 的失败模式
  • Unit Tests for 已知 Bug — 每发现一个 Bug，立刻写成测试
  • 人工标注数据 — 30-50 条高质量 Pass/Fail + Critique
❌ 避免
  • 测试所有东西 — 维护成本极高，反而降低速度
  • 通用指标 — BERTScore、ROUGE 等与真实业务脱节
  • 提前设计 Eval 框架 — 没有数据之前，框架是空架子
  • 过度自动化 — 有些失败模式用人工就够了
📊 时间分配参考

Hamel 的经验：AI 产品项目中，60-80% 的开发时间花在 Error Analysis 和 Eval 上（主要是看数据、理解失败），而不是构建自动化检查。如果你大多数时间在搭架子而不是看数据，方向可能偏了。

Q4：如果我没有数据或客户，我该从哪里开始？

没有数据是最常见的起点困境。正确答案是：先创造数据，而不是先建 Eval 体系。

1
尽快上线，哪怕是最小版本

给 5-10 个内测用户（内部同事、友好用户），让他们真实使用。真实用户会以你完全预料不到的方式使用产品，这是任何合成数据都无法替代的。

2
用合成数据覆盖已知场景（谨慎使用）

没有真实用户时，可以用 LLM 生成合成的用户输入，然后把这些输入喂给你真实的 AI 系统，观察实际回答。只合成用户输入，不合成 AI 回答——你要测试的是你自己的系统。

3
自己扮演用户，做 5-10 次完整对话

带着"我是用户"的心态和你的系统交互，记录所有让你觉得"不太对"的地方。这些就是你第一批 Eval 的种子。最小可行 Eval：20-50 条输出 + 人工 Pass/Fail 判断。

⚠️ 合成数据的边界

合成数据无法揭示你不知道自己不知道的失败模式（unknown unknowns）。LLM 总会以意想不到的方式失败，只有真实用户数据才能暴露这些盲区。

应该跟踪哪些指标？

✅ 有意义的指标

  • 业务专属 Pass Rate — 基于 Error Analysis 发现的真实失败类别
  • 用户投诉率 / 负面反馈率 — 最直接的真实信号
  • 重试率 — 重试说明上一个答案没满足需求
  • 关键场景覆盖率 — 最重要场景的通过情况

❌ 避免依赖

  • BERTScore / ROUGE — 和用户满意度没有可靠相关性
  • 通用 LLM 评分 — 不了解你业务标准的 LLM 打分是噪音
  • 整体平均分 — 掩盖具体失败模式，无法指导行动
  • Likert 量表 — 分数差别意味着什么？行动不明确
💡 指标设计原则

每个指标都应该能回答："如果这个指标变好/变差了，我下一步做什么？" 答不上来的指标对你没有价值。

应该使用哪些工具？

工具是次要的，流程和原则是第一位的。工具选择不当的代价是：用工具的限制约束了你的思维。

最重要
数据查看器（Data Viewer）

能快速查看 Trace 全貌的工具。强烈推荐用 Notebook（Jupyter）自己搭，1 天内可完成。通用平台往往无法展示你业务最关键的上下文信息。

次重要
Trace 日志记录

把每次交互的完整 Trace 记下来。工具不重要，哪怕写进数据库或 CSV 都行。关键是有，不是用什么框架。

按需
自动化 Eval 框架

等你确认了值得自动化的失败模式，再考虑用什么框架。工具应该服务于你已经想清楚的 Eval 设计，而不是反过来。

Q6：我可以实现测试和评估的自动化吗？如果可以，我如何才能信任它？

可以，但"信任自动化 Eval"需要主动建立，它不会自然而然地存在。

🔑 核心原则：自动化 Eval 必须以人工标注为锚点

没有人工建立的 Ground Truth，自动化 Eval 对齐的是什么？它只是在放大随机噪音。人工标注是所有自动化 Eval 的根基，这个根基不能省略。

1
建立校准数据集：30-50 条人工 Pass/Fail + Critique

让 Benevolent Dictator 对这些 Trace 做出判断并写下详细理由。这些数据是验证 LLM Judge 准不准的"测试集"。

2
用校准数据验证你的自动化 Eval

把校准数据集喂给 LLM Judge，看它的判断和人工判断的一致率是多少。通常 >85% 才有理由信任。一致率低，先改 Judge，再说自动化。

3
Critique Shadowing：让 LLM Judge 学习人工判断逻辑

把专家的 Critique 作为 LLM Judge 的 Few-shot 示例，让 LLM 学习同样的判断逻辑。这是对齐 LLM Judge 最有效的方式。

4
持续定期抽样验证，不要一劳永逸

每隔一段时间（如每两周），随机抽 20-30 条 Judge 的判断和人工判断对比，确认一致率没有下滑。产品变化、用户行为变化都可能让 Judge 漂移。

✅ 适合自动化
  • 已知失败的回归测试（Unit Tests）
  • 格式和结构验证（正则/断言）
  • 安全性检查（有害内容、越界请求）
  • 高频且有足够校准数据的判断任务
⚠️ 谨慎自动化
  • 复杂的多维度质量判断
  • 需要专业知识的领域判断
  • 校准数据少于 30 条的维度
  • 结果会直接触发高影响行动的判断

❌ 错误。正确答案是B。主观性是 Eval 必须解决的问题，不是回避的理由。多人投票引入的是噪音，工具无法解决标准模糊的问题。
第 2 题 · 场景题
产品还没有真实用户，你应该先做什么？
A. 设计完整的 Eval 框架，等用户来了再填数据
B. 尽快让 5-10 个内部同事真实使用，收集最小量真实 Trace，同时用合成数据覆盖已知场景
C. 用 LLM 生成 1000 条合成对话，建立完整数据集
D. 等有 100 个付费用户再开始做 Eval
✅ 正确！先创造真实数据，哪怕只有 5-10 个内部用户。真实用户会以意想不到的方式使用产品，合成数据覆盖不了这些盲区。最小可行 Eval：20-50 条真实 Trace + 人工标注。
❌ 错误。正确答案是B。等待完美数据集是最常见的拖延。框架是空的，要先有数据才能填充。1000 条合成数据不如 20 条真实数据。
第 3 题 · 判断
只要 LLM Judge 的一致率在建立时验证过一次，后续就可以持续信任它。
A. 正确

🌊 三个海湾模型（Three Gulfs）— 课程核心框架

这是整门课的核心诊断框架，用来回答"我的 AI 系统为什么出问题？"三个 Gulf 分别对应三类根本性挑战。

理解的海湾（Gulf of Comprehension）

开发者 ↔ 数据之间

你对数据的理解总是不完整的：真实用户 Query、系统输出、合成测试用例。你只能建立一个局部模型，永远无法完全掌握系统真正在做什么。

真实案例

Cleric 构建生产事故分析 AI 时，工程师常说"看起来不错，但我不确定是否正确"——这就是 Gulf of Comprehension：没有 Ground Truth，你无法判断输出好坏。

解法：系统性 Error Analysis；构建自定义数据查看器；看足够多的 Trace 达到理论饱和。

规范的海湾（Gulf of Specification）

开发者 ↔ LLM Pipeline 之间

你在努力描述自己想要什么，但大量意图是隐性的——只有看到不满意的输出，你才意识到自己其实想要什么。Prompt 永远无法完全捕捉你的真实意图。

真实案例

Sherlock 的 API 错误信息规范失败：只给 AI Agent 返回 "HTTP 422"，不给详细错误字段。Agent 陷入死循环。一旦暴露完整错误描述，Agent 立即修正行为。

解法：Pass/Fail + Critique 循环；用具体失败案例改进 Prompt；明确 LLM Agency Level（低/中/高 自主性）。

泛化的海湾（Gulf of Generalization）

数据 ↔ LLM Pipeline 之间

即使你把一切都规范清楚了，模型在遇到训练分布之外的数据时仍然会失败。这是模型本身的局限，需要不同的策略——Fine-tuning、任务分解、RAG 改进。

真实案例 1

Discord 的 Clyde 聊天机器人在长对话或创意 Jailbreak（"grandma jailbreak"）时会"出戏"，无法保持一致的人格——这就是泛化失败。需要持续监控 + 安全护栏。

真实案例 2

Cleric 的奇特发现：高级模型在测试环境中会检测到"它在模拟中"（通过 API 响应时间、数据分布等微信号），从而改变行为——这是泛化到了意料之外的环境特征。

🔄 分析-测量-改进循环（Analyze → Measure → Improve）

课程把所有内容组织在这个三阶段循环里。每个阶段都有经典的反模式需要避开。

① 分析（Analyze）
② 测量（Measure）
③ 改进（Improve）

目标：定性理解，建立失败分类体系

用 Open Coding → Axial Coding，把模糊的"感觉不对"转化成可命名、可计量的失败类别。核心步骤：

  • 准备数据集（真实 Trace 或维度采样合成数据）
  • Open Coding：逐条阅读，贴描述性标签
  • Axial Coding：把标签聚类成结构化分类
  • 反复迭代直到理论饱和（新数据没有新类别）
❌ 分析阶段的常见陷阱
  • 把标注外包给不懂业务的人 — 技术上正确但业务上失败的回答，只有领域专家能识别
  • 看的案例不够多 — 没有达到理论饱和就停止，会遗漏重要失败模式
  • 跳过分析阶段 — Shreya 强调：无论公司大小，Error Analysis 永远是必须做的第一步
💡 维度采样法（Dimensional Sampling）

没有真实用户时，用这个方法生成高质量合成数据：①识别有意义的变化维度（用户意图、用户画像、查询复杂度）→ ②在各维度上生成组合 → ③把每个组合转成真实 Query。避免"程序化生成的千篇一律"问题。

目标：量化失败频率，为改进提供优先级依据

两类 Eval 工具，不同场景用不同的：

程序化 Eval（首选）

快、便宜、客观。能用代码检查的，不要用 LLM。

  • 检查 Tool 是否被调用 → 直接看 tool object 是否有值
  • 正则匹配、Schema 验证、关键词检测
  • 对于 Agent：分别检查各阶段（工具选择、参数生成、执行结果、输出处理）
LLM Judge（主观/复杂 → 才用）

处理主观或需要推理的判断，但必须先建立信任。

  • 用 Train/Dev/Test 切分标注数据集（不能泄露！）
  • 用 Dev 集迭代改 Judge Prompt
  • 用 Test 集计算 TPR/TNR（测量 Judge 可靠性）
  • 用 judgy 库做 Bias Correction
❌ 测量阶段的常见陷阱
  • LLM Judge 没有对齐人工判断 — 没有人工数据作锚点的 Judge 是放大噪音
  • 数据泄露 — 把失败 Trace 放进 Judge Prompt，再用这些 Trace 测试 → 自我实现预言
  • TNR 低 = 虚假信心，TPR 低 = 低估系统能力；两个都要测量

目标：做最小干预，解决真实的、已被量化的问题

改进策略应该按成本从低到高尝试，不要一上来就上大招：

低
Prompt 工程

大多数情况下，改 Prompt 就足够了。结构化提示：静态内容放前面（最大化缓存命中，省约 90% Token 成本），动态内容放后面。批量请求额外省 50%。

中
架构调整

任务分解（把一个复杂 LLM 调用拆成多个聚焦的步骤）、修复 Tool 描述、改进检索策略（RAG）。

高
Fine-tuning 和 Model Cascade

Model Cascade：基于查询复杂度在便宜模型和贵模型之间路由。前提是置信度 ↔ 正确性之间有足够相关性（生成式任务未必成立）。Fine-tuning 是最后手段，先确认是 Gulf of Generalization 问题再考虑。

❌ 改进阶段的常见陷阱
  • 跳过分析和测量，直接改进 — 最常见的浪费时间方式
  • 直接跳到 Fine-tuning — 80% 的情况下 Prompt 工程就够了，但很多人"听说过 Fine-tuning"就想直接上

Transition Failure Matrix（Agent 系统）

对于多步骤 Agent 系统，用状态转移失败矩阵找调试优先级：哪个状态转移失败率最高？那就是优先修的地方。

评估的分层抽样策略

每周目标：约 100 条 Trace。三种采样混合：

  • 随机采样：整体健康检查
  • 不确定性采样：LLM Judge 置信度低的用例（最有学习价值）
  • 失败驱动采样：违规、Bug、用户投诉触发的用例

评估条件漂移（Evaluation Criteria Drift）

标注过程中，你对失败模式的理解会演变。标注结束时的你和开始时的你，看数据的眼光已经不同了——这意味着你必须回去重新标注早期数据，直到标准真正一致。这是正常的，不是 Bug。

LLM Agency Levels

必须在 Prompt 里明确指定 LLM 的自主程度：高自主（主动重新解读用户意图）/ 中等（提出澄清问题）/ 低自主（字面执行用户请求）。不同产品场景对应不同选择，影响用户如何与系统交互。
